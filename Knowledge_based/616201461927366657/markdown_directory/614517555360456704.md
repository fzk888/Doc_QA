                                   3.RT优化\
                                   背景\
                                   报价单在后端系统内存基于多叉树数据结构，角色查看报价单以后端的数据结构发送给前端展示。在数据有1万节点时，RT性能为＞５分钟。\
                                   \
                                   问题：DB的二维结构到内存的多叉树结构消耗cpu时间\
                                   优化：后端系统多叉结构改为简单的Map结构，从开发复杂度下降和调试变得简单，提高查看和修改效率。原来消耗io和cpu变成io；\
                                   \
                                   问题:后端系统组装好多叉树结构以Json形式发送给前端。后端cpu资源稀缺，没有利用用户端cpu资源，不敏感数据。\
                                   优化:后端发送随机的报价单数据，前端负责随机数据转换成多叉树结构，减轻后端cpu消耗。\
                                   \
                                   问题：报价单的价格受汇率影响，每次从DB获取数据都要计算产品，模型，价格.１万节点的报价单，计算时间长，RT不达标。\
                                   a优化:相同的报价单因不同的时间汇率不同影响报价单价格计算。价格规则计算消耗cpu资源，计算时间过长，DB提取一个报价单计算一次，用户的最终需要的是最后一个价格计算，为了提高RT，自动计算调整为用户决定什么时间计算价格。\
                                   b优化：在审批场景中，报价单读＞写。价格计算的数据持久化，由动态数据静态化。\
                                   c优化：单个报价单节点数据1万时，节点计算总次数也有１万。串行计算１万次，计算总时间接受不了。用线程池进行并行计算，计算总时间减少，主线任务程保存上下文场景和非阻塞。\
                                   \
                                   问题:报价单本地缓存还是中央缓存\
                                   a优化:报价单创建状态阶段,报价体读和写偏重，命中率不高，如果采用本地缓存，水平扩展不好；从excel导入报价单，最大的文件有7M左右(几十或上百万条数据)，中央缓存redis也不好。报价单创建阶段，使用数据库保持一致性。\
                                   b优化:报价处于审批阶段，报价单读＞写。报价单数据＜1000，报价单转化成json字符串存入中央缓存redis中；报价单＞1000时，采用本地缓存，减少DB读压力。\
